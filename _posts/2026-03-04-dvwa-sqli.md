### **Introduction**

In this lab, we analyze the **SQL Injection** vulnerability in Damn Vulnerable Web Application (DVWA) across three security levels: **Low, Medium, and High**.

The application accepts a user ID as input and queries the database to retrieve the corresponding first and last name from the `users` table. For example, when ID `1` is submitted, the system returns the record for the user **admin admin**. However, due to improper input handling in certain security levels, this functionality becomes vulnerable to SQL injection.

At the **Low** security level, user input is directly inserted into the SQL query without any sanitization. This allows attackers to manipulate the query logic, retrieve all records, enumerate database tables and columns, and even extract usernames and password hashes.

At the **Medium** security level, basic protections such as `mysqli_real_escape_string()` and restricted input methods are introduced. Although these measures improve security, they can still be bypassed by intercepting and modifying HTTP requests.

At the **High** security level, the application uses session-based input to retrieve user data. However, if session values are not properly validated, they may still be manipulated, potentially leading to SQL injection through tampered session parameters.

This lab demonstrates how improper input validation, weak filtering mechanisms, and insecure handling of user-controlled data can lead to serious vulnerabilities such as authentication bypass and sensitive information disclosure.

### **Security: High**
After analyzing the source code, we can observe that the application uses the session ID value as part of the query to retrieve data from the users table.

This means that if we are able to manipulate the session ID, we may influence how the backend query is constructed and executed.

By clicking on “here to change your ID”, a new window opens that allows us to modify our session ID value. Since this value is later used in the database query without proper validation or sanitization, it introduces a potential attack vector where the session parameter can be tampered with to manipulate the query logic.
